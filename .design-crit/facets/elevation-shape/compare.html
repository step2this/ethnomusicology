A Open in new tab
Flat Crate
Why This Option
Maximum flatness. Zero shadows anywhere. All depth via background tint steps and 1px borders. Matches the "mostly flat" directive from the locked component design as literally as possible. DJ software (Rekordbox, Serato, Traktor) uses this approach — flat rows, flat headers, scrim for overlays.
Works Well For
Dense data tables with hundreds of rows. Zero rendering overhead. Simplest possible token set (3 surface tints, 4 radius values, no shadow tokens). Fastest to implement.
Watch Out For
The slide-over panel relies solely on scrim + tint for depth. Sort dropdowns and tooltips may not feel elevated enough — borders alone may not separate them from the table rows beneath. Toast could blend with content surface on some monitors.
B Open in new tab
Wire Lift
Why This Option
Flat table + targeted shadows on overlays. The daily-use catalog view is completely shadow-free (fast, dense, clean). Shadows appear only on transient elements that genuinely float: the import slide-over (directional shadow), toasts, sort dropdown, and the primary CTA button. The rule is simple: "if it overlaps content, it gets a shadow."
Works Well For
Products where the primary view is a dense data table but overlays need physical presence. The slide-over gets a convincing "sliding in" effect with a leftward directional shadow. The primary button shadow is a subtle affordance cue. 3 shadow tokens, 4 surface tints — low complexity.
Watch Out For
The shadow/no-shadow boundary needs to be clear to developers: inline = flat, floating = shadow. Not hard, but needs to be documented. Otherwise, this is a very safe option with no real risk.
C Open in new tab
Soft Stack
Why This Option
Full Material-style depth. Every surface above L0 casts a shadow, creating a smooth depth gradient from the flattest row to the most prominent overlay. 5 levels with doubling blur progression. Adds an 8px "md" radius token for floating elements.
Works Well For
Applications that want rich, tangible layering. The 5-level system gives maximum flexibility for future complexity. The "rounder = floating" radius convention is a nice touch for overlay elements.
Watch Out For
Contradicts the "mostly flat" directive from component design. Shadows on every track row creates visual noise with 100+ rows. L1 and L2 shadows are nearly indistinguishable on dark backgrounds. More tokens to maintain for marginal benefit. Moves toward consumer SaaS, away from DJ-software aesthetic.
My Take
Go with B (Wire Lift)

A is pure, B is practical, C is too much.


A (Flat Crate) is the most faithful to the DJ-software reference and the "mostly flat" component design. It's a valid choice. But the import slide-over — the most important overlay in the product — benefits from a real shadow. Scrim alone says "something is on top." Scrim + directional shadow says "a panel slid in from the right." That physical cue matters for a transient panel the DJ uses dozens of times.


C (Soft Stack) is over-built. Shadows on track rows contradicts the locked component design, adds visual noise in a 100-row catalog, and pushes toward a consumer SaaS feel that doesn't match the user's established DJ-software preferences.


B (Wire Lift) is the sweet spot: flat where you browse (the daily table view), shadowed where you act (import panel, toasts, primary CTA). Simple rule: "inline = flat, floating = shadow." 3 shadow tokens, 4 surface tints, 4 radius values. Clean, fast, implementable, and it gives the slide-over the physical presence it deserves.


The shape language is a non-decision — all three options use the same 4-token radius scale (0/3/6/full) already locked by the component design. The real question is just: how many things get shadows?
