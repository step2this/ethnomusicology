Purchase Link Panel (UC-020)

Expandable panel on track tiles showing search links to DJ stores (Beatport, Bandcamp, Juno Download, Traxsource) for purchasing tracks discovered through LLM-generated setlists.

Brief → Facets → Direction
Platform Web (Flutter)
Stack Flutter / Dart / Riverpod / Material 3
Scope Component (within existing track tile)
Brief confirmed. Next up: component design facet for the purchase panel.
This is a component-level brief within the existing Ember Crate design system. The full-app design direction is already locked.

Target Users & Context

DJs at a laptop during set prep. They've just generated or reviewed a setlist and want to buy tracks -- especially LLM suggestions not in their catalog. The purchase action is secondary to browsing/listening but critical for converting discovery into acquisition.

Core Interaction Loop

See track → want to buy → expand purchase panel → tap store link → store opens in new tab with search results. One tap per store, collapsed by default.

Differentiator

Multi-store search links from a single panel. Most DJ tools link to one store. This gives DJs choice across four stores covering mainstream electronic and underground/independent catalogs.

Scope

In v1
  • Expandable purchase panel on each track tile (collapsed by default)
  • 4 store links: Beatport, Bandcamp, Juno Download, Traxsource (fixed order)
  • Lazy-loaded on expand via API call
  • Store icon/button treatment per store
  • url_launcher to open in external browser tab
  • Clipboard fallback if url_launcher fails
  • Empty state when both title and artist are missing
Out of v1
  • Hit verification (checking if store has the track)
  • Store preference reordering
  • Bulk purchase links for entire setlist
  • Affiliate tag registration
  • Custom store icons/logos

Existing Design Language

Full design system locked from previous crit cycle: Ember Crate color palette (amber/copper on warm blacks), Inter Tight + JetBrains Mono typography, 4px spacing grid, Wire Lift elevation (flat table, shadows on floats), Studio density (40px rows).

Open Design Questions

#QuestionImpact
1 Expandable inline panel vs bottom sheet? Inline keeps context but causes layout shift. Bottom sheet avoids shift but loses position context. Layout, animation, mobile
2 Store icon treatment -- Material icons, first-letter avatars ("B", "J"), or simple text buttons? No official store logos as Flutter icons. Component design, density
3 Trigger button placement -- end of metadata chip row, near play button, or new row below metadata? Hierarchy, scannability

Decisions from Brief

DecisionChoiceRationale
Interaction Collapsed-by-default expandable panel Avoids visual clutter. Lazy API call on expand.
Store count 4 stores, fixed order Beatport > Bandcamp > Juno > Traxsource. Covers mainstream + underground.
Visual separation Separate from source attribution Attribution = where found. Purchase = where to buy. Different concerns.
Empty state Store search page handles it Always show all 4 links. Store handles "not found."
Link opening url_launcher + clipboard fallback External browser tab. SnackBar on clipboard fallback.
