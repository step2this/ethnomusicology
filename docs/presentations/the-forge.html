How the Forge works

What Is the Forge?

Just markdown. No plugins, no SDKs, no special tooling.

  agents/ ← team roles

Each command file is a prompt template that casts Claude into a specialized role:

  • Requirements Architect — writes use cases
  • Devil's Advocate — finds gaps
  • Task Planner — builds dependency graphs
  • Quality Gate — verifies postconditions
  • Blind Reviewer — grades against spec

The command header is the entire secret:

---
description: Review for gaps and untestable postconditions
allowed-tools: Read, Glob, Grep
---

Tool restriction = behavioral constraint.
allowed-tools: Read, Glob, Grep means it can find problems but can't fix them.

This is the entire technology. A markdown file with a YAML header that restricts which tools Claude can use, and a body that assigns a persona and step-by-step instructions. You can write your first command in 5 minutes.

/uc-create

Requirements as prompt engineering — Cockburn use cases from conversation

8 interactive dialog groups, each a structured prompt:

  1. Classification (goal level, priority)
  2. Actors (primary, supporting)
  3. Conditions (pre, post, invariants)
  4. Main Success Scenario (happy path)
  5. Extensions — MOST CRITICAL
  6. Variations
  7. Agent Execution Notes
  8. Acceptance Criteria

Each postcondition → a test assertion
Each extension → error handling code

The single most important line in this project:

## Extensions

- 8c. LLM hallucinates catalog tracks
      (returns track_ids that don't exist):
  1. Backend validation catches non-existent
     track_ids
  2. Those tracks are reclassified as
     'suggestion'
  3. Continues with corrected data

Without this extension, the system would happily serve hallucinated track IDs to the frontend. The devil's advocate review caught this before a single line of code was written.

The Extensions section is explicitly marked "MOST CRITICAL" in the uc-create command. Error paths get more attention than happy paths. This is where the Forge pays for itself — every extension you identify here is a bug you never have to debug in production.

/uc-review

Adversarial review before code exists

---
description: Review for gaps and untestable
  postconditions
allowed-tools: Read, Glob, Grep
---

# Review Use Case: $ARGUMENTS

You are a **Devil's Advocate** reviewing
a Cockburn-style use case.

## Step 7: Extension Coverage (MOST CRITICAL)

For EACH MSS step:
  → What if it fails?
  → Unexpected input?
  → Dependency unavailable?
  → Race condition?
  → Wrong permissions?

Key design choice: tool restriction

allowed-tools: Read, Glob, Grep

The devil's advocate cannot Write. It can find problems but can't fix them. This forces it to report issues rather than silently patching.

ST-003 review results

14 issues found before implementation:
→ Missing retry logic
→ Unspecified migration DDL
→ No test for hallucinated track IDs
→ No enforcement of daily rate limits

All fixed before a single line of code.

The tool restriction is the key insight. When Claude can only read, it becomes a genuine adversary — it has to report what's wrong instead of quietly fixing it. This produces a complete issue report that you review before implementation begins.

/task-decompose

From use case to parallel task graph

## ST-001 Dependency Graph

T1 (migration 003) ──┐
                      ├── T3 (DB query)
T2 (AppError nested) ─┤      │
                      │      ├── T5 (route)
                      │      │      │
                      │      │      ├── T6 (integration test)
T4 (response types) ──┘      │      │
                             │      └── T7 (frontend screen)
                             │             │
                             │             └── T8 (widget test)

Critical path: T1 → T3 → T5 → T6
Parallel:      Backend (T1-T6) ‖ Frontend (T7-T8)

Each task specifies:

  • Module path — backend/src/routes/tracks.rs
  • Acceptance criteria — testable assertions
  • Blocked by / Blocks — dependency chain
  • Size — S / M / L / XL
  • Risk — Low / Medium / High
  • Agent assignment — non-overlapping files

Why this matters

The dependency graph determines which tasks can run in parallel. T1 and T2 have no dependencies — two agents can start simultaneously. The critical path tells you exactly where bottlenecks are.

This is the command that enables multi-agent teams. Without the dependency graph, you don't know what can run in parallel. With it, you can assign agents to non-overlapping tasks and let them work concurrently.

/grade-work

Grade the code like you'd grade a paper — blind review against spec

CategoryWeightWhat It Measures
Postcondition Coverage30%Does every postcondition have a test?
Extension Handling25%Is every error path implemented AND tested?
Invariant Enforcement15%Enforced by types or assertions?
Code Quality15%Zero warnings, no panics, doc comments
Test Quality15%Unit + integration, clear naming

Verdict Scale

A (90-100%) → APPROVED
B (80-89%)  → APPROVED
C (70-79%)  → CONDITIONAL
D (60-69%)  → REJECTED
F (<60%)    → REJECTED

Bonuses & Penalties

+5% Doc comments on all public items
+5% Property-based tests
-10% Per unwrap() in production code
-15% Panics on recoverable errors

This is the closest thing to a code review rubric that's actually machine-executable. The blind reviewer pattern means it evaluates code ONLY against what the spec says — not what the code looks like. It can't be swayed by clean formatting or clever comments.

Steel Threads & Spikes

De-risking integration before building features

The Problem with Use Cases Alone

13 use cases, zero proof they connect

After implementing UC-001 (Spotify Import), the project had 12 more UCs planned. But:

  • Each UC is self-contained — none proves layers connect
  • Does Beatport actually return BPM in Camelot format? Unknown
  • Can Flutter play audio cross-origin? Unknown
  • Does essentia fit in a 512MB container? Unknown

We needed two new tools:

  • Spikes — time-boxed experiments to reduce uncertainty
  • Steel threads — thin vertical slices that prove integration end-to-end

Use Case = Feature Depth

UC-016: 12 MSS steps, 15 extensions, 9 postconditions
Covers every edge case of setlist generation
But doesn't prove Frontend → API → Claude → DB

Steel Thread = Integration Proof

ST-003: Thin slice across 5 layers
8 integration assertions
Proves the connection works end-to-end

Spike = Hypothesis Test

SP-001: "Does Beatport return BPM?" — 4 hours
Binary answer with evidence
De-risks before committing

This is when we introduced the walking skeleton methodology. Use cases define WHAT to build. Steel threads prove HOW it connects. Spikes answer WILL IT WORK. Three different tools for three different questions.

Spikes — Falsifiable Hypotheses

3 spikes, 12 hours total, saved weeks of rework

## Hypothesis

Beatport v4 API provides authenticated access
to track metadata including BPM, musical key,
and genre, with sufficient rate limits for
batch import workflows.

## Timebox: 4 hours

  → OAuth 2.0 with public client_id workaround

Q2: BPM and key returned?
  → YES. Native fields.

Q3: What format?
  → BPM: integer (128)
  → Key: raw notation ("A♭ min")
  → NOT Camelot format natively

## Decision: Partially confirmed
  ✅ BPM numeric, key native
  ⚠️ Needs Camelot lookup table
  ⚠️ Rate limits undocumented

What spikes prevent

SP-002 discovered CORS would block audio streaming. Without the spike, this would have been found mid-implementation of the crossfade preview feature — after days of coding.

SP-003 proved the container needs 1-2GB, not 512MB. The infrastructure decision changed before anyone wrote a Dockerfile.

Every spike has a falsifiable hypothesis, a timebox, specific questions, and a binary decision. "Partially confirmed" is a valid result — it means "proceed with adjustments." The spike format forces rigor: you have to answer each question with evidence, not vibes.

"I Wrote It So I Think It's Fine"

The self-review blind spot — from the actual retrospective

Retrospective findings (verbatim)

1. Lead implemented solo
Plan specified 4 parallel agents. Lead wrote all 9 tasks. Violated "lead coordinates, does not implement" directive.

2. No independent code review
Agent reviewed its own code. Self-review misses the same class of issues the writer created.

3. Pre-commit hook was half-built
Only backend checks. Flutter analyze + test were NOT enforced. Frontend code was unguarded.

4. Directives without enforcement
CLAUDE.md said "always use teams" but nothing prevented the lead from ignoring it.

"Directives are not enforcement. 'Always use teams' written in CLAUDE.md doesn't prevent a solo run. You need either hooks (automated enforcement) or workflow gates (human checkpoints) to make it real."
PatternImpact
Context rot in long sessionsHigh
Self-review blind spotHigh
Pure-function modules ideal for delegationMedium
Directives ignored without enforcementHigh
This is the key learning: the process failed because directives were not enforcement. Writing "always use teams" in a markdown file doesn't prevent a solo run. The ST-003 retrospective forced us to evolve the Forge — adding hooks, mandatory critic agents, and stronger imperatives.

The Fix — Process Evolution

5 changes applied within 1 hour of the retrospective

## Agent Teams

+ > **MANDATORY: The lead agent MUST NOT implement code directly.**
+ > The lead coordinates, reviews, wires modules together,
+ > and runs quality gates.
+ > All implementation code MUST be written by spawned subagents.

+ > **MANDATORY: Use multi-agent teams for any task touching
+ > >2 files or >100 lines.**

### Code Review: Critic Agent (REQUIRED)

+ > **MANDATORY: Every implementation MUST have a separate critic
+ > review before /verify-uc.** The critic agent runs in a
+ > FRESH context — it has NOT watched the code being written.

## hooks/PreToolUse:
- command: "cd backend && cargo fmt --check && cargo test"
+ command: "cd backend && cargo fmt --check && cargo clippy &&
+ cargo test && cd frontend && flutter analyze && flutter test"

The /retrospective command didn't just document what went wrong — it generated specific action items and applied them immediately. The process improved itself.

Five changes in one hour. Blockquoted imperatives are harder to ignore than regular text. The pre-commit hook now runs all 5 quality gates. And the Critic Agent requirement means no code reaches the verify step without a fresh-context review.

The Critic Agent Pattern

Fresh context breaks the self-review blind spot

Builder Agent (context rot)

  Wrote camelot.rs      ✓
  Wrote arrangement.rs  ✓
  Wrote setlist.rs      ✓
  Wrote frontend        ✓
  "Looks fine to me"

─────────────────────────────

Critic Agent (fresh context)

  Reads git diff main...HEAD cold
  Has NOT watched code being written
  "Line 47: wrong crate name"
  "Line 203: stale variable"
  "Tests missing edge case"

How it works

  1. Builders complete their work
  2. Spawn a Critic Agent in fresh context
  3. Critic reads git diff main...HEAD, the plan, and test output
  4. Critic looks for:
    Missed edge cases, dead code, unused imports, naming inconsistencies, plan deviations, context-rot artifacts
  5. Critic sends feedback with file:line references
  6. Only after critic approves → /verify-uc
"The generator-critic separation is essential. The agent that writes code should not be the only one reviewing it."
The key insight is information asymmetry. The builder has been staring at this code for hours — it can't see what it already "knows." The critic sees the diff for the first time. It catches a fundamentally different class of bugs. This is the same principle behind why code review works for humans.

Multi-Agent Teams

Coordination patterns that actually work

Team Architecture

Lead coordinates, never implements. Builders own non-overlapping directories.

Lead (Coordinator)
Never writes implementation code
Creates tasks, reviews, wires modules

┌──────────┬──────────┬──────────┐
│          │          │          │
Backend-1  Backend-2  Frontend
src/api/   src/services/ lib/
src/db/    camelot.rs    screens/
         arrangement.rs

             │
         Critic Agent
     Fresh context, reads diff cold

Coordination Rules

  • Only ONE agent edits a given file at a time
  • Define shared traits up front so parallel agents produce compatible interfaces
  • Integration/wiring modules (main.rs, routes) stay with the lead
  • Flag blockers immediately via messaging
// Shared trait — defined before agents start
pub trait ImportRepository {
    async fn upsert_track(&self, track: &Track)
        -> Result;
}

// Backend-1 implements this in src/db/
// Backend-2 consumes it in src/services/
// No coordination needed — the trait IS the contract
The key pattern is non-overlapping file ownership. Backend-1 owns src/api and src/db. Backend-2 owns src/services. They communicate through shared traits defined before implementation begins — the trait is the contract. This is the same principle as API contracts but at the module level.

Pure Functions Are Delegation Gold

camelot.rs: 420 lines, 17 tests, zero coordination overhead

/// Compute Camelot compatibility score between keys.
/// Same key = 1.0, adjacent = 0.9,
/// ±2 = 0.5, else = 0.0
pub fn camelot_score(
    a: &CamelotKey,
    b: &CamelotKey
) -> f64

/// BPM proximity score.
/// |a-b| mapped to 1.0 (0-2) through 0.0 (>10)
pub fn bpm_score(a: f64, b: f64) -> f64

/// Position-aware energy arc score.
/// Build 0-40%, peak 40-75%, cooldown 75-100%
pub fn energy_arc_score(
    energy: i32,
    position: usize,
    total: usize
) -> f64

/// Combined transition score.
/// Camelot 50% + BPM 30% + Energy 20%
pub fn transition_score(...) -> f64

Why pure functions delegate well

✓ No IO, no state, no network calls
✓ Deterministic — same input, same output
✓ Fully testable in isolation
✓ Zero coordination with other agents
✓ Clear spec: scoring rules are the entire prompt

"Ralph Wiggum Loops"

For test-heavy pure modules, consider autonomous loops: 

while true; do
  claude "implement and pass all tests"
done

Quality comes from iteration + automated test feedback, not self-assessment. The test suite IS the acceptance criteria.

Result

camelot.rs: 17 tests, 0 fix iterations (fresh agent)
arrangement.rs: 10 tests, 0 fix iterations

If your codebase has pure-function modules, delegate them. They're the highest-ROI use of multi-agent teams. Zero coordination overhead, fully testable, and the spec is the test suite itself. camelot.rs had 17 tests and compiled on the first try when a fresh agent wrote it — compared to 5+ fix iterations when the same logic was part of a longer session.

Quality Gates That Actually Gate

Pre-commit hook in settings.json — the agent cannot bypass it

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash(git commit.*)",
        "hooks": [
          {
            "type": "command",
            "command": "cd backend && cargo fmt --check
              && cargo clippy -- -D warnings
              && cargo test
              && cd ../frontend && flutter analyze
              && flutter test",
            "timeout": 300000
          }
        ]
      }
    ]
  }
}

5 gates, 1 hook

1. cargo fmt --check ← formatting
2. cargo clippy -D warnings ← lint
3. cargo test ← 135 tests
4. flutter analyze ← frontend lint
5. flutter test ← 12 tests

Directives vs. Enforcement

Directive: "Always run tests before committing"
→ Agent can ignore this. And did (ST-003).

Enforcement: Pre-commit hook blocks git commit
→ Agent cannot commit without all 5 gates passing.

One is a suggestion. The other is a constraint.

This is the lesson from ST-003: directives are suggestions, hooks are enforcement. The pre-commit hook runs all 5 quality gates before every single commit. The agent cannot bypass it. This is what turned the "always run tests" directive from a hope into a guarantee.

Write Your First Slash Command

A slash command is just a markdown file — 5 minutes to write

# File: .claude/commands/review.md

---
description: Review a design doc for gaps
allowed-tools: Read, Glob, Grep
---

# Review: $ARGUMENTS

You are a **Devil's Advocate** reviewing a design document.

## Step 1: Load the Document
Read the file at $ARGUMENTS.

## Step 2: Check for Untestable Requirements
For each requirement, ask:
"How would I write an automated test for this?"
Flag anything that can't be tested automatically.

## Step 3: Check Error Paths
For EACH step in the happy path:
  → What if this fails?
  → What if input is unexpected?
  → What if a dependency is unavailable?

## Step 4: Generate Report
Output findings as:
  CRITICAL — must fix before implementation
  WARNING  — should fix, creates risk
  SUGGESTION — nice to have

Format: [SEVERITY] [Section] Description

Run with: /review docs/my-design-doc.md

This is the entire technology. YAML frontmatter with a description and tool restrictions, then a body with a persona, numbered steps, and an output format. The $ARGUMENTS variable is whatever the user types after the slash command. Five minutes to write, and it runs adversarial review on any document.

Universal Patterns

Applicable to any codebase, any team, any AI coding tool

01

Specification before implementation.
Use cases and steel threads as executable specs. Don't write code until the error paths are documented.

02

Adversarial review before code.
Devil's advocate on requirements, not just code review. Catch bugs before they exist.

03

Dependency graphs before parallelism.
Know what blocks what before spawning agents. The graph IS the coordination.

04

Fresh-context critic.
The generator should not be the only reviewer. Different context catches different bugs.

05

Hooks > directives.
Automated enforcement beats README instructions. If you can't hook it, it's a suggestion.

06

Retrospectives update tooling.
Process changes applied to .claude/ files, not just meeting notes. The process evolves.

07

Design decisions as data.
design-tokens.json, not Figma screenshots. Machine-readable, version-controlled, traceable.

08

Pure functions delegate best.
No IO, no state, deterministic = zero coordination overhead. Test suite IS the acceptance criteria.

These eight patterns transfer to any team, any codebase, any AI coding tool. They're not specific to Claude Code — they're software engineering patterns amplified by AI assistance. The ones that will have the most immediate impact: adversarial review, hooks over directives, and the fresh-context critic.
